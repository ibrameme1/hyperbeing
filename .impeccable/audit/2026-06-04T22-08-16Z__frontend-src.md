## Detailed Findings by Severity

### [P1] Icon-only buttons missing aria-label
- **Location**: `Dashboard.jsx` — theme toggle button (~line 829), AccountMenu trigger button (~line 838), delete/trash button (~line 331)
- **Category**: Accessibility
- **Impact**: Screen readers announce these as "button" with no context. A user with a screen reader cannot distinguish "theme toggle" from "account" from "delete."
- **WCAG**: 1.1.1 Non-text Content, 4.1.2 Name, Role, Value
- **Recommendation**: Add `aria-label` to every icon-only button. Theme toggle: `aria-label={isDark ? 'Switch to light mode' : 'Switch to dark mode'}`. Account button: `aria-label={`Account menu for ${user?.name}`}`. Delete button: `aria-label={`Delete ${pres.title}`}`.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P1] Touch targets below 44px minimum
- **Location**: `Dashboard.jsx` — delete button `w-7 h-7` (28px); confirm-delete buttons `px-2 py-1 text-[10px]` (~24px tall); attachment remove button `w-5 h-5` (20px); aspect ratio buttons `px-2.5 py-1` (~28px tall)
- **Category**: Responsive Design / Accessibility
- **Impact**: On touch devices, users frequently mis-tap small targets. The confirm-delete popup is especially risky: tapping Cancel when intending Delete (or vice versa) on a 24px button in a 28px-total card is error-prone. Especially bad because Delete is irreversible.
- **WCAG**: 2.5.5 Target Size (AAA), Apple HIG 44pt minimum
- **Recommendation**: Increase all interactive targets to min 44×44px. Delete button: `w-11 h-11`. Confirm-delete buttons: `px-3 py-2 text-xs`. Attachment remove: `w-6 h-6` minimum, or position it as a tappable overlay with larger hit area using padding.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P1] HeroSection.jsx gradient text — missed in prior fix pass
- **Location**: `HeroSection.jsx:76`
- **Category**: Anti-Pattern
- **Impact**: `bg-gradient-to-b from-white via-white/95 to-white/70 bg-clip-text text-transparent` is the same AI-tell pattern removed from Homepage.jsx. If HeroSection is shown to users, this undoes the previous fix.
- **Recommendation**: Replace with `text-white` — the fade to transparency provides no legibility benefit and is the definition of decorative gradient text.
- **Suggested command**: `$impeccable quieter HeroSection.jsx`

### [P1] Credit ring: color-only status signal
- **Location**: `Dashboard.jsx` — `AccountMenu` component, `ringColor` calculation (~line 358)
- **Category**: Accessibility
- **Impact**: The SVG credit ring cycles green → amber → red as credits deplete. Color is the sole differentiator. Users with protanopia or deuteranopia (8% of males) cannot distinguish green from red. The ring is the primary low-credit warning mechanism.
- **WCAG**: 1.4.1 Use of Color
- **Recommendation**: Add a secondary signal: change the ring dash pattern (solid → dashed when low), or show a text label inside/below the ring ("Low" when `pct ≤ 20`). This pairs well with the existing `low && <p>Running low</p>` text already inside the dropdown.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P2] No ARIA `role="dialog"` on blocking overlays
- **Location**: `Dashboard.jsx` — `AnalyzingOverlay` (~line 52), `fetchingUrls` overlay (~line 747), `creatingPresentation` overlay (~line 773)
- **Category**: Accessibility
- **Impact**: All three overlays use `fixed inset-0 z-50` and block all interaction, but have no `role="dialog"`, no `aria-modal`, no `aria-label`, and no focus management. Screen readers don't announce the overlay as a dialog, and keyboard focus remains on the background content.
- **WCAG**: 4.1.2 Name, Role, Value; 2.1.1 Keyboard; 2.1.2 No Keyboard Trap (inverse)
- **Recommendation**: Add `role="dialog" aria-modal="true" aria-label="Nova is analyzing your brief"` (or equivalent) to each overlay container. On mount, move focus to the first focusable element in the dialog.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P2] No ARIA live region for generation status
- **Location**: `Dashboard.jsx` — `AnalyzingOverlay` cycling messages; `PresentationPage.jsx` — streaming progress
- **Category**: Accessibility
- **Impact**: The speech bubble messages cycling through Nova's thoughts ("ok let me read through this…") are invisible to screen readers. A user relying on a screen reader gets no feedback during the 5-10 second analyzing wait.
- **WCAG**: 4.1.3 Status Messages
- **Recommendation**: Add `aria-live="polite" aria-atomic="true"` to the speech bubble message container. The messages cycle every 2.4 seconds which is readable aloud without interrupting the user.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P2] Off-brand `#764ba2` accent in Moodboard AttachZone
- **Location**: `Dashboard.jsx:1007` — `accentColor="#764ba2"` on the Moodboard `AttachZone`
- **Category**: Theming
- **Impact**: `#764ba2` is not a brand color. `#8B5CF6` (hb-primary) should be used. Minor visual inconsistency.
- **Recommendation**: Change `accentColor="#764ba2"` to `accentColor="#8B5CF6"` (or `accentColor="#007AFF"` like the Branding zone uses a blue accent — consider using two distinct brand colors to differentiate the zones).
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P2] `transition: height` on eye blink animation
- **Location**: `Dashboard.jsx:90` — `transition: 'height 0.08s ease'` in robot eye style
- **Category**: Performance
- **Impact**: Animating `height` triggers browser layout on every frame. The 0.08s duration minimizes this, and the element is small, so real-world impact is low but measurable on low-end devices.
- **Recommendation**: Replace with `scaleY` transform: `transform: blink ? 'scaleY(0.2)' : 'scaleY(1)'`, `transition: 'transform 0.08s ease'`. Same visual effect, GPU-composited.
- **Suggested command**: `$impeccable optimize Dashboard.jsx`

### [P2] Stacked backdrop-blur overlays
- **Location**: `Dashboard.jsx` — `AnalyzingOverlay` and `creatingPresentation` overlay both use `backdropFilter: blur(18px)` and `backdropFilter: blur(16px)` on full-screen overlays
- **Category**: Performance
- **Impact**: Full-screen `backdrop-filter: blur` is expensive on mobile, especially on older devices. Two overlays are unlikely to stack simultaneously, but each individually taxes the compositor.
- **Recommendation**: Use `background: rgba(0,0,0,0.85)` without blur for the outermost overlay — users won't see the background through the blur anyway since the card is centered. Reserve blur for the card itself if needed.
- **Suggested command**: `$impeccable optimize Dashboard.jsx`

### [P2] Email input missing `<label>` in HeroSection
- **Location**: `HeroSection.jsx:112` — `<input type="email">` with dynamic placeholder, no `<label>` element
- **Category**: Accessibility
- **Impact**: Placeholder text is not a substitute for a label. Screen readers announce the input as unlabeled when it receives focus. The dynamic typewriter placeholder also disappears immediately on focus.
- **WCAG**: 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions
- **Recommendation**: Add a visually hidden `<label htmlFor="hero-email">Your email</label>` and `id="hero-email"` on the input.
- **Suggested command**: `$impeccable harden HeroSection.jsx`

### [P2] `PresentationCard` has no keyboard navigation
- **Location**: `Dashboard.jsx:243` — card is a `motion.div` with `onClick` but no `role="button"` or `tabIndex`
- **Category**: Accessibility
- **Impact**: Keyboard-only users cannot tab to or activate presentation cards. The grid is entirely inaccessible without a mouse.
- **WCAG**: 2.1.1 Keyboard
- **Recommendation**: Either change the outer `motion.div` to `motion.article` with `tabIndex={0} role="button" onKeyDown={e => e.key === 'Enter' && navigate(...)}`, or wrap with `<Link>`.
- **Suggested command**: `$impeccable harden frontend/src/pages/Dashboard.jsx`

### [P3] Aspect ratio buttons have no descriptive label for screen readers
- **Location**: `Dashboard.jsx:888-900` — `['16:9', '4:3', '1:1', '9:16']` buttons
- **Category**: Accessibility / Usability
- **Impact**: "16:9" means nothing to a non-technical first-time user. A screen reader user hears "16:9 button, 4:3 button" with no context.
- **Recommendation**: Add `aria-label` like `aria-label="16:9 widescreen"`. For sighted users, a tooltip (`title`) already works.
- **Suggested command**: `$impeccable clarify frontend/src/pages/Dashboard.jsx`

### [P3] Missing `loading="lazy"` on presentation card images

- **Impact**: All card thumbnails load eagerly, even those below the fold. With 12+ cards, this is ~12 network requests on mount.
- **Recommendation**: Add `loading="lazy"` to all card thumbnail images.
- **Suggested command**: `$impeccable optimize frontend/src/pages/Dashboard.jsx`

### [P3] `shimmer` keyframe defined twice

## Patterns & Systemic Issues

**Touch targets are consistently too small.** The 44px minimum is violated throughout: delete button (28px), confirm-delete buttons (24px), attachment remove button (20px), aspect ratio buttons (~28px). This is a systemic pattern — the whole app was designed for desktop and touch was not a primary consideration.

**ARIA is treated as optional.** No icon-only button across the Dashboard has an `aria-label`. No overlay has a dialog role. No status messages use live regions. This is not a few oversights — it's a missing layer of implementation.

**Gradient text was partially cleaned up but not globally.** The `HeroSection.jsx` component was outside the scope of the previous quieter pass and still carries the gradient H1. A global grep for `bg-clip-text` would catch any remaining instances.

## Positive Findings

1. **Token system is solid.** The CSS custom property system (`--bg-page`, `--text-primary`, etc.) is well-structured. Most product UI uses `var(--token)` correctly. Dark mode is consistent in the product app.
2. **Skeleton loaders are excellent.** The shimmer animation on the presentation grid, both the aspect-ratio div and text lines, is exactly right — it gives users a structural preview of what's loading.
3. **Error messages are plain-language and specific.** The `submitError` handling distinguishes 429 (rate limit), 413 (payload too large), and 402 (out of credits) with actionable copy. This is above average.
4. **Session cache + SSE pattern is smart.** Loading from `sessionStorage` immediately while the HTTP fetch is in flight eliminates perceived latency. The SSE updates are additive, not replacing everything.
5. **The `confirmDelete` two-step pattern is correct.** The inline approach (first click → show Delete/Cancel) avoids a modal and is the right UX pattern for this action.
